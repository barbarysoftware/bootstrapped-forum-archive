You can implement your own in about 2 weeks time.

The advantages of implementing your own:

  • Much more freedom in tailoring it for your needs.
  • Save $$
  • The thrill of learning new things. I enjoy learning techy stuff.

Broadly, the idea is the following:

Compute some kind of signature (aka ‚Äúmachine ID‚Äù, ‚Äúmachine fingerprint‚Äù) for the computer. Use things like MAC Address, CPU, disk etc.

When your software is first run, check whether ‚Äúcached license‚Äù (stored somewhere on the user‚Äôs machine) is present. (More on that below). If not, go into first-time activation.

  • First time activation: Get the registration key from the user. Send the registration key and machine signature to your server. At the server, check whether the software with
    that key has already been installed on stipulated number of machines. If so, send ‚Äúrejection‚Äù. Else, allow activation and send back encrypted form of the machine signature back.
  • The encrypted machine signature is received at the client. Store it somewhere on the user‚Äôs machine. This is what I referred to as the ‚Äúcached license‚Äù.

Each time the software runs, read the ‚Äúcached license‚Äù which in essence is the
encrypted machine signature. Decrypt the cached license. Compare the decrypted machine signature with the actual signature. If they are ‚Äúnearly same‚Äù (i.e., ‚ÄúFuzzy Match‚Äù), accept the validity of installation.

The reason for checking that they are ‚Äúnearly same‚Äù instead of ‚Äúexactly same‚Äù is that you don‚Äôt want to reject the cached license when the user does small changes to the hardware. How much change you want to tolerate is up to you.

To further clarify the idea, the server side program does the following:

  • Listens to activation requests from the software running on user‚Äôs
    machine. As I mentioned, the activation request payload is essentially
    "Registration key & machine signature".
  • Maintains a simple database/file that keeps a map of ‚Äúregistration key‚Äù to ‚ÄúSignatures of machines on which it has already been installed‚Äù.
  • Check whether the machine signature ‚Äúmatches‚Äù(fuzzy match) one of the machines
    software with that key has already been installed. If so, it is a case
    of user trying to re-install on a machine he had previously installed.
    So, send back ‚ÄúAccepted payload‚Äù.
  • Otherwise, check whether allowing to be installed on the new machine is within the limits of the license. If so, send back ‚ÄúAccepted payload‚Äù. Else, send back ‚ÄúRejected
    payload‚Äù.
  • The ‚ÄúAccepted payload‚Äù is in its essence the encrypted
    version of the machine signature that came in as part of activation
    request.
